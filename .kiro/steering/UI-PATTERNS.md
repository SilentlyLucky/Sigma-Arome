# UI Patterns & Known Pitfalls

Rules learned from implementation bugs. Follow these to avoid repeating mistakes.

---

## 1. SelectDropdown â€” Never Pass Empty Choices

**Problem:** Buildpad `SelectDropdown` throws "Choices option configured incorrectly" when `choices` is an empty array `[]`.

**Rule:** Always guard `SelectDropdown` with a length check. If choices are empty or still loading, render a disabled `Input` with an informative placeholder instead.

```tsx
// âťŚ WRONG â€” crashes when choices is []
<SelectDropdown label="Product" choices={productChoices} value={value} onChange={onChange} />

// âś… CORRECT â€” guard against empty choices
{loading ? (
  <Input label="Product" placeholder="Loading..." disabled onChange={() => {}} />
) : choices.length > 0 ? (
  <SelectDropdown label="Product" choices={choices} value={value} onChange={onChange} />
) : (
  <Input label="Product" value="" placeholder="No items available" disabled onChange={() => {}} />
)}
```

## 2. FK UUID Fields â€” Never Auto-Disable

**Problem:** DaaS sets `default_value: "gen_random_uuid()"` on all UUID columns including FK fields. Buildpad's `hasAutoGeneratedDefault()` then marks them as readonly â†’ dropdown disabled.

**Rule:** `hasAutoGeneratedDefault()` and `isUUIDAutoGenerated()` must NEVER return true for relation fields (M2O/O2M/M2M) or fields with `foreign_key_table`. Already patched in `lib/buildpad/field-interface-mapper.ts`.

**Prevention:** After creating any new collection with UUID FK fields, always run:
```
mcp_daas_fields update â†’ schema.default_value: null
```
on every FK field.

## 3. Schema default_value â€” Never Use as Input Value

**Problem:** PostgreSQL stores system defaults like `''::character varying`, `gen_random_uuid()`, `now()` in `schema.default_value`. If passed directly to form inputs, users see garbage text.

**Rule:** Never use `field.schema.default_value` directly as a form input value. Use `getFieldDefault()` which filters out:
- Function calls: `gen_random_uuid()`, `now()`, `nextval()`
- Type casts: `::character varying`, `::text`
- Data type names: `uuid`, `integer`, `text`
- Empty string casts: `''::character varying`

---

## 4. DaaS System Tables â€” Permission Required for Non-Admin

**Problem:** Non-admin users get 403 on system tables (`daas_relations`, `daas_fields`, `daas_permissions`, `daas_user_roles`) which Buildpad components call directly from browser.

**Rule:** When creating a new operational role, always grant read permission on:
- `daas_relations` â€” needed by M2O/O2M dropdown hooks
- `daas_fields` â€” needed by CollectionForm/FieldsService
- `daas_permissions` â€” needed by PermissionsService
- `daas_user_roles` â€” needed by role detection (with `user_id = $CURRENT_USER` filter)

---

## 5. Readonly Fields in Create Mode â€” Show Preview

**Problem:** Auto-generated fields (order_number, batch_number) are `readonly: true`. In create mode, they show empty disabled inputs with no context.

**Rule:** For readonly fields in create mode, show "(Auto-generated on save)" as the value. Implemented in `FormField.tsx`:
```tsx
if (context === 'create' && field.meta?.readonly) {
  return '(Auto-generated on save)';
}
```

## 6. M2O Dropdown Labels â€” Human-Readable

**Problem:** FK fields display as "Material Id", "Supplier Id" with UUID values.

**Rule:** Every M2O field must have:
1. `translations: [{ language: "en-US", translation: "Material" }]` â€” for form label
2. `options.template: "{{name}} ({{code}})"` â€” for dropdown display
3. `options.enableCreate: false` â€” unless the user's role owns that master data

## 7. Middleware â€” Minimize Network Calls

**Problem:** Calling `getUser()` + `getSession()` + DaaS fetch on every request makes navigation slow.

**Rule:**
- `updateSession()` already calls `getUser()` â€” don't call it again
- Parse Supabase cookie directly for user ID and token (zero network calls)
- Cache role lookups for 10 minutes
- Add 2-second timeout on DaaS fetch
- Skip role enforcement for `/`, `/login`, `/api/*`, `/auth/*`

---

## 8. React DOM Props â€” No Lowercase `readonly`

**Problem:** DaaS metadata uses lowercase `readonly`. React DOM requires camelCase `readOnly`. Spreading raw metadata into components causes React warnings.

**Rule:** In `FormFieldInterface.tsx`, always pass `readOnly` (camelCase). In components that receive `...rest` props, destructure and remove lowercase `readonly` before spreading.

---

## 9. Status Fields â€” Backend-Controlled, Hidden from Form

**Problem:** Users manually changing status bypasses workflow rules.

**Rule:**
- Status fields: `hidden: true` in form, `readonly: true`
- Status changes only via action buttons on detail pages
- Backend extensions validate transitions (reject invalid state changes)
- UI shows status as Badge, not as editable dropdown

---

## 10. Auto-Number Fields â€” Sequential with Date Prefix

**Format:** `{PREFIX}-{YYYYMMDD}-{0001}`

**Rule:** Auto-number extensions must:
1. Query existing records with today's prefix
2. Sort descending, limit 1
3. Parse last segment as integer
4. Increment by 1
5. Pad to 4 digits

Never use random numbers for business document numbers.

## 11. Production Order State Machine

### States

    draft â†’ planned â†’ released â†’ in_progress â†’ completed

    Off-ramps: any active state â†’ on_hold (resumable) or cancelled (terminal)

### Role Authority

| Role               | Allowed transitions                                   |
|--------------------|-------------------------------------------------------|
| PPIC               | draftâ†’planned, plannedâ†’released                       |
| System (auto)      | plannedâ†’released / waiting_issue, waitingâ†’released    |
| Warehouse Operator | releasedâ†’in_progress (after full material issue)      |
| Production         | in_progressâ†’completed                                 |
| Any                | *â†’on_hold, *â†’cancelled                                |

### Key Rules

- PPIC releases; Extension B auto-runs the stock check. If any material is short, status becomes `waiting_issue` automatically.
- Extension C watches for batches becoming `stored_available` and re-runs the check; orders auto-release when all shortages clear.
- Extension A (validator) enforces both the transition map AND role authority for every status change.
- Warehouse Operator: `released â†’ in_progress` is blocked until all `material_request_items.issued_qty >= requested_qty`.
- Production: only `in_progress â†’ completed`. Cannot start production (WO does that).
